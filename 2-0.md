# AstroMonitor 2.0 — TUI Upgrade Plan

## Overview

Version 2.0 replaces the CLI flag-driven interface with an interactive Terminal UI (TUI). The binary becomes a self-contained guided experience: it detects whether the user has configured a token, walks them through first-time setup if not, and then presents an action dashboard.

---

## Technology Choices

| Concern | Choice | Rationale |
|---|---|---|
| TUI framework | `ratatui` (0.26+) | Active successor to `tui-rs`; widely used in the Rust ecosystem |
| Event loop / input | `crossterm` | Cross-platform terminal backend; already the default for `ratatui` |
| JSON config | `serde` + `serde_json` | Standard; lightweight; no async needed |
| CLI parsing | **removed** | No flags needed; everything is driven through the TUI |

Remove `structopt` from `Cargo.toml` once CLI parsing is no longer needed.

---

## Config File Specification

**Path:** `~/.config/astromonitor/astro.json`  
(resolved via `dirs::config_dir()`, already available through the `dirs` crate)

```json
{
  "token": "<telegram bot token>"
}
```

This replaces passing the token on the command line. The `Paths` struct in `src/lib.rs` gains a `config_file_path` field pointing to this file.

## Phase 1 — Project Scaffolding & Config Layer

**Goal:** Lay the structural groundwork without breaking the existing build.

### Steps

1. **Add dependencies to `Cargo.toml`**
   - `ratatui = "0.26"`
   - `crossterm = "0.27"`
   - `serde = { version = "1", features = ["derive"] }`
   - `serde_json = "1"`

2. **Create `src/config.rs`**
   - Define `AppConfig { token: String }` with `serde::Serialize/Deserialize`
   - `fn config_path() -> PathBuf` — returns `~/.config/astromonitor/astro.json`
   - `fn load_config() -> Option<AppConfig>` — reads and deserializes the file; returns `None` on missing or malformed file
   - `fn save_config(config: &AppConfig) -> Result<(), String>` — creates the directory if needed, writes the JSON file

3. **Extend `Paths::init()` in `src/lib.rs`**
   - Add `pub config_file_path: PathBuf` using `dirs::config_dir()`

4. **Verify the build still compiles** — no behaviour change yet.

## Phase 2 — TUI Shell & Application State Machine

**Goal:** Establish the event loop and the screen-switching state machine.

### Application States

```
AppState::Boot
    │
    ├─ config exists ──► AppState::Dashboard
    │
    └─ config missing ──► AppState::Setup(SetupStep::Instructions)
                               │
                               └─► AppState::Setup(SetupStep::TokenEntry)
                                        │
                                        └─► AppState::Setup(SetupStep::Confirm)
                                                 │
                                                 ├─ confirmed ──► (save) ──► AppState::Dashboard
                                                 └─ cancelled ──► AppState::Setup(SetupStep::TokenEntry)
```

### Steps

1. **Create `src/tui/mod.rs`** — module root, re-exports public items

2. **Create `src/tui/app.rs`**
   - `enum AppState` with the variants listed above
   - `struct App { state: AppState, token_input: String, config: Option<AppConfig> }`
   - `impl App { fn new() -> Self }` — calls `load_config()` to determine initial state
   - `fn handle_key(&mut self, key: KeyEvent)` — central input dispatcher; delegates to the active screen handler

3. **Create `src/tui/runner.rs`**
   - `fn run() -> Result<(), Box<dyn Error>>` — initialises `crossterm` raw mode, creates a `ratatui::Terminal`, runs the main event loop, restores terminal on exit
   - The loop calls `terminal.draw(|f| ui::render(f, &app))` then `app.handle_key(event)`

4. **Update `src/main.rs`**
   - Remove all `structopt` / `CliArgs` usage
   - Call `tui::runner::run()` instead

## Phase 3 — Setup Flow (First-Run Experience)

**Goal:** Guide a new user from zero to a saved token.

### Screen: Instructions (`SetupStep::Instructions`)

- Full-screen panel explaining:
  - "No configuration found."
  - "To use AstroMonitor you need a token from the Telegram bot."
  - Steps: open Telegram → search `@AstroMonitorBot` → send `/register` → copy the token
- Single action: **[Press Enter to continue]**

### Screen: Token Entry (`SetupStep::TokenEntry`)

- A labelled single-line text input: `Enter your token:`
- Character-by-character input; Backspace to delete
- `Enter` to advance; `Esc` to go back to Instructions
- Basic non-empty validation before advancing

### Screen: Confirm (`SetupStep::Confirm`)

- Shows the entered token (partially masked if desired)
- Two highlighted buttons: **[Confirm]** / **[Cancel]**
- Arrow keys or Tab to switch focus; Enter to activate
- On Confirm: call `save_config()`; transition to `AppState::Dashboard`
- On Cancel: return to Token Entry

### Implementation files

- `src/tui/screens/setup.rs` — render logic for all three setup steps
- `src/tui/screens/mod.rs` — re-exports

## Phase 4 — Dashboard Screen

**Goal:** The main screen a configured user sees on every launch.

### Layout

```
┌─────────────────────────────────────────────────┐
│           AstroMonitor 2.0                      │
├─────────────────────────────────────────────────┤
│                                                 │
│      ┌─────────────────┐                        │
│      │   Take Backup   │  ← focused             │
│      └─────────────────┘                        │
│                                                 │
│      ┌─────────────────┐                        │
│      │ Restore Backup  │                        │
│      └─────────────────┘                        │
│                                                 │
├─────────────────────────────────────────────────┤
│  [↑↓] Navigate   [Enter] Select   [q] Quit      │
└─────────────────────────────────────────────────┘
```

### Behaviour

- Arrow keys move focus between buttons
- Enter activates the focused button; triggers the corresponding backup operation (calls existing `backup::database` functions)
- A status/result bar at the bottom shows success or error messages after each operation
- `q` or `Esc` quits cleanly

### Implementation files

- `src/tui/screens/dashboard.rs` — render and input handling for the dashboard
- The actual backup calls remain in `src/backup/database.rs` — no logic moves

## Phase 5 — In-Progress & Result Feedback

**Goal:** Show the user what is happening during backup/restore (these are blocking HTTP calls).

### Steps

1. Add `AppState::Working { label: String }` — renders a spinner/progress message while the operation runs
2. Run the backup/restore on the same thread (keeping the no-async rule); draw a static "Working…" frame before the call, draw the result frame after
3. Add `AppState::Result { message: String, success: bool }` — shows outcome; any key returns to Dashboard

---

## Phase 6 — Polish & Cleanup

1. **Remove legacy CLI code** — delete `CliArgs`, remove `structopt` from `Cargo.toml`
2. **Remove `--fd-monitor` and `--system-monitor` flags** — these were advanced/debugging features; decide whether to surface them behind a hidden key shortcut in the TUI or drop entirely for 2.0
3. **Update `CLAUDE.md`** — reflect new architecture, new dependencies, removed dependencies
4. **Update `README.md`** — rewrite usage section; remove all `--flag` examples; describe the TUI flow
5. **Update `install.sh`** if needed — no functional changes expected
6. **Bump version** in `Cargo.toml` to `2.0.0`
7. **Update CI** (`.github/workflows/build-check.yml`) — binary name and artifact naming unchanged; verify cross-compilation still works with new deps

## Out of Scope for 2.0

- KStars monitoring (`--kstars`) — background monitoring in a TUI requires threading or async; defer to 2.1
- `--fd-monitor` / `--system-monitor` — defer to 2.1 as optional TUI panels
- Configuration editing from within the TUI (token change) — defer to 2.1
